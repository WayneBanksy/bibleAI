## 1. Algorithm Choice Rationale

### Why AES-256-GCM

AES-256-GCM (Advanced Encryption Standard, 256-bit key, Galois/Counter Mode) was selected
for message content encryption because:

| Property | AES-256-GCM | Alternatives rejected |
|----------|-------------|-----------------------|
| **Authenticated encryption** | Yes — GCM provides both confidentiality and integrity via a 128-bit authentication tag. Tampering is detected on decryption without a separate HMAC step. | AES-CBC requires a separate HMAC (Encrypt-then-MAC) to detect tampering, introducing more moving parts. |
| **NIST standard** | NIST SP 800-38D. Used in TLS 1.3, AWS KMS default, Google Cloud default. | Non-standard ciphers carry ecosystem and audit risk. |
| **Performance** | Hardware-accelerated (AES-NI) on all modern server CPUs. GCM is parallelisable. | ChaCha20-Poly1305 is a valid alternative but less universal in HSM/KMS integrations. |
| **Key size** | 256-bit key: maximum AES key size; exceeds NIST minimum (128-bit) for post-quantum margin. | 128-bit is sufficient today but 256-bit costs nothing at modern hardware speeds. |
| **Library support** | `cryptography.hazmat.primitives.ciphers.aead.AESGCM` — the standard Python cryptography library (used by pyca). | `pycryptodome` would work but adds a dependency when `cryptography` is already required. |

**Summary:** AES-256-GCM is the industry-standard choice for envelope encryption of
application data. It provides authenticated encryption (AEAD), is NIST-approved, has
native CPU acceleration, and is widely supported by cloud KMS providers for future HSM
integration.

## 2. Key Derivation Design

### Goal

Each user's messages are encrypted with a distinct 256-bit key. This provides **per-user
key isolation**: compromising one user's key does not expose any other user's messages.

Keys are **derived on demand** (not stored), reducing the attack surface: the database
contains only encrypted ciphertext. The only secret that must be protected is the single
`MASTER_KEY_SECRET` environment variable.

### HKDF-SHA256

Key derivation uses **HKDF** (HMAC-based Key Derivation Function, RFC 5869) with SHA-256.

```
MASTER_KEY_SECRET (env var, bytes)
        │
        ▼
  HKDF-SHA256
  ┌──────────────────────────────────────────┐
  │  IKM  = MASTER_KEY_SECRET                │
  │  salt = user_id.bytes  (16 bytes, UUID)  │
  │  info = b"bible_therapist_message_v1"    │
  │  length = 32 bytes                        │
  └──────────────────────────────────────────┘
        │
        ▼
  per-user 256-bit AES key  (never stored)
```

**Why HKDF:**
- HKDF is the NIST-recommended KDF for expanding a shared secret into multiple
  domain-specific keys (RFC 5869, NIST SP 800-56C).
- It is deterministic: same inputs always produce the same key, so the key can be
  re-derived at any time without storage.
- The `info` context label (`b"bible_therapist_message_v1"`) domain-separates these keys
  from any other keys derived from the same master secret.
- The `user_id` as salt ensures per-user key isolation even if two users have the same
  message content.

**Implementation:** `cryptography.hazmat.primitives.kdf.hkdf.HKDF`

## 3. Nonce Management

### Why a random 12-byte nonce per encryption

AES-GCM security depends on **nonce uniqueness**: if the same (key, nonce) pair is ever
used twice, GCM's confidentiality and integrity guarantees collapse.

The design uses a **random 96-bit (12-byte) nonce** generated by `os.urandom(12)` on
every encryption call. This is the NIST-recommended nonce length for GCM (NIST SP 800-38D
§8.2.1) and avoids the counter-management complexity of deterministic nonces.

### Collision probability analysis

A random 96-bit nonce has a collision probability of approximately:

```
P(collision) ≈ n² / (2 × 2^96)
```

For a given user key, a collision requires **two encryptions to produce the same 12-byte
nonce**. Given:

| Scenario | Messages per user | Collision probability |
|----------|-------------------|-----------------------|
| MVP (1 year active user) | ~10,000 | < 10⁻²¹ |
| Extreme power user (10 years) | ~1,000,000 | < 10⁻¹⁷ |

These probabilities are negligible for MVP volumes. If message volumes approach 2³² per
user key, switch to a deterministic counter-based nonce with a nonce registry.

### Blob format

The nonce is **prepended to the ciphertext** in the stored BYTEA blob:

```
bytes:  [ nonce (12) | ciphertext (variable) | auth_tag (16) ]
```

The Python `cryptography` AESGCM implementation appends the auth tag to the ciphertext
automatically. The `decrypt` function splits the blob at byte 12 to recover the nonce
before decryption.

## 4. MASTER_KEY_SECRET Management

### What it is

`MASTER_KEY_SECRET` is a high-entropy secret string from which all per-user keys are
derived. It must be treated with the same care as a private key or database credential.

### Generating a value

```bash
openssl rand -base64 32
# Example output: 7Kd9Qx+mNpW8hL3fRvZoYeJ1uXcB6gTiAs4MnD5kEw=
```

### How it must be provided

| Environment | Method | Notes |
|-------------|--------|-------|
| **Production** | Secret manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault, etc.) | Never hardcode. Injected as env var at container startup. |
| **Staging/CI** | Secret manager or CI secrets (GitHub Actions encrypted secrets, etc.) | Separate value from production; never share between environments. |
| **Local dev** | Falls back to built-in weak default (`dev-insecure-do-not-use-in-production`) | Clearly labelled insecure. App refuses to start with this value in `production` mode. |

### Application startup behaviour

`config.py` validates `MASTER_KEY_SECRET` at startup:

- **`environment=development`**: Uses the weak default if `MASTER_KEY_SECRET` is not set.
  This allows the test suite and local docker-compose to run without secrets.
- **`environment=production`** (or any non-development environment): Refuses to start if
  `MASTER_KEY_SECRET` is missing or equals the insecure default. Raises `ValueError`
  with a clear message during Pydantic settings validation.

### What must never happen

- `MASTER_KEY_SECRET` must never appear in:
  - Application logs (structured or plaintext)
  - Error messages returned to clients
  - Sentry / error tracker payloads
  - Git history (not in `.env`, `docker-compose.yml` production configs, or source code)
- Derived per-user keys must also never be logged.

## 5. Rotation Runbook (Stub)

> **Note:** This is a planning stub. A full automated re-encryption migration script is
> a post-MVP deliverable. The steps below document the procedure so it can be planned,
> reviewed, and scripted before the first rotation is needed.

### When to rotate

- `MASTER_KEY_SECRET` has been (or is suspected to have been) exposed.
- Scheduled rotation per security policy (e.g., annually).
- Migrating to a new key derivation scheme.

### Rotation steps

1. **Generate the new key**
   ```bash
   openssl rand -base64 32
   # Store as MASTER_KEY_SECRET_NEW in the secret manager. Do NOT yet
   # replace MASTER_KEY_SECRET_OLD.
   ```

2. **Deploy a dual-read application version**
   - Add `MASTER_KEY_SECRET_NEW` alongside `MASTER_KEY_SECRET_OLD` to the running
     environment.
   - Deploy an application version that, on decryption failure with the current key,
     falls back to attempting decryption with the new key (to handle in-flight
     re-encryptions). This version does NOT write with the new key yet.

3. **Run the re-encryption migration**
   ```
   # Pseudocode — implement as a one-off Alembic script or standalone migration tool
   for each message in messages:
       plaintext = decrypt(old_key_derived_for(message.user_id), message.content_encrypted)
       message.content_encrypted = encrypt(new_key_derived_for(message.user_id), plaintext)
   commit()
   ```
   - Run during a low-traffic window or as a background job with rate limiting.
   - Checkpoint progress so the script is resumable if interrupted.

4. **Swap the active key**
   - Set `MASTER_KEY_SECRET` = `MASTER_KEY_SECRET_NEW` in the secret manager.
   - Remove `MASTER_KEY_SECRET_OLD` from the environment.
   - Deploy the final application version (single-key mode).

5. **Verify**
   - Spot-check a random sample of messages: attempt decrypt with the new key.
   - Run the integration test suite against the rotated database.

6. **Revoke the old key**
   - Delete `MASTER_KEY_SECRET_OLD` from the secret manager.
   - Record the rotation event in the security audit log.

### Downtime considerations

With step 2 (dual-read deployment) and a background migration script, a rotation can be
performed **without downtime**. The migration script processes messages at a controlled
rate to avoid database lock contention.

## Appendix: Blob Layout Diagram

```
Stored in messages.content_encrypted (BYTEA):

 byte 0          byte 11    byte 12          byte N-17  byte N-16        byte N
 ┌───────────────────────┬──────────────────────────────┬────────────────┐
 │  nonce  (12 bytes)    │  ciphertext  (variable len)  │  auth_tag (16) │
 └───────────────────────┴──────────────────────────────┴────────────────┘
```

The authentication tag is generated and verified by AES-GCM automatically.
`decrypt()` raises `ValueError` if the tag does not match (tamper-evident).
